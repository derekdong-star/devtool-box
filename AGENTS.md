## 提交规则

### 触发词：「提交」

当我说**提交**时，**不要立即执行 git 操作**，而是先对当前待提交内容进行 Review，流程如下：

1. 执行 `git diff` 和 `git status`，获取完整变更内容
2. 从以下三个维度进行审查：

   **架构整洁**
   - 分层是否清晰（handler / service / model 职责有无越界）
   - 是否有重复代码、冗余逻辑
   - 错误处理是否统一、是否有被静默丢弃的 error
   - 命名是否一致（变量、函数、文件）

   **扩展性**
   - 新增功能是否依赖具体类型而非接口
   - 模块注册方式是否符合现有约定（在 `app.go` 的 `modules` 追加）
   - 硬编码的魔法数字、字符串是否应提取为常量

   **功能**
   - 前后端接口是否对齐（请求/响应格式、字段名）
   - 边界条件是否处理（空值、非法输入、并发）
   - 前端 JS 语法是否正确（可用 `node --check` 验证）
   - 是否影响已有功能

3. 给出 Review 结论：
   - 若发现问题：列出问题清单，提出优化方案，**等待确认**
   - 若无问题：说明"未发现问题，可以提交"，**等待确认**

---

### 触发词：「确认提交」

当我说**确认提交**时，执行以下操作：

1. 用一句话总结本次变更的核心内容（作为 commit message）
2. 执行：
   ```
   git add .
   git commit -m "<总结内容>"
   git push
   ```
3. 推送完成后，输出 commit hash 和变更文件数量
